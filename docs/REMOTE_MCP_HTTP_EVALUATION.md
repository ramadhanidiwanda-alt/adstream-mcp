## MCP SDK v1.29.0 Support

### Transports Available

| Transport | Class | Import Path |
|---|---|---|
| Stdio | `StdioServerTransport` | `@modelcontextprotocol/sdk/server/stdio.js` |
| SSE | `SSEServerTransport` | `@modelcontextprotocol/sdk/server/sse.js` |
| Streamable HTTP | `StreamableHTTPServerTransport` | `@modelcontextprotocol/sdk/server/streamableHttp.js` |
| Web Standard Streamable HTTP | `WebStandardStreamableHTTPServerTransport` | `@modelcontextprotocol/sdk/server/webStandardStreamableHttp.js` |
| Express | `createMcpExpressApp()` | `@modelcontextprotocol/sdk/server/express.js` |

### Streamable HTTP Details

- **Session ID**: Required in stateful mode (generated by `sessionIdGenerator`), optional in stateless mode.
- **GET/POST separation**: GET for SSE stream, POST for JSON-RPC messages, DELETE for session teardown.
- **Auth**: Handled externally via standard HTTP auth (Authorization header, middleware).
- **Existing MCP tools reuse**: Yes — tools registered via `Server.setRequestHandler(CallToolRequestSchema, ...)` remain unchanged.
- **Stdio default**: Yes — stdio can remain the default. Streamable HTTP would be an opt-in transport.

### Upgrade Path

#### Breaking Changes (SDK v0.5.0 → v1.29.0)

1. **Import paths** — Old paths (`@modelcontextprotocol/sdk/server/index.js`, `@modelcontextprotocol/sdk/types.js`) still resolve via wildcard export.
2. **`Server` class deprecated** — `@deprecated` in favor of `McpServer` high-level API. Still functional.
3. **`ServerOptions`** — `capabilities` changed from required to optional. Both forms compile.
4. **New zod dependency** — SDK v1.29.0 requires `zod@^3.25 \|\| ^4.0`. Current project uses `zod@^3.23.8`.
5. **Heavy new dependencies** — `express@^5.2.1`, `hono@^4.11.4`, `jose@^6.1.3`, `cors@^2.8.5`, `express-rate-limit@^8.2.1`, `ajv@^8.17.1`, `@hono/node-server@^1.19.9`, etc.
6. **Node version** — Requires Node >= 18 (current: v24.14.1, OK).
7. **TypeScript build config** — Current tsup config should be compatible; no changes expected.

#### Scope of Upgrade

- Update `mcp-server/package.json` SDK version
- Update `package.json` root `zod` version if needed (to satisfy peer dep)
- Update import statements in `mcp-server/src/index.ts`, `mcp-server/src/createServer.ts`
- No changes needed to tool registration or handler APIs
- `npm install` to fetch 10+ new transitive dependencies

## Feasibility Result

**Status: SDK v1.29.0 supports official Streamable HTTP transport.**

However, the upgrade is a **medium-scope change** with many new transitive dependencies and potential compatibility issues with zod version requirements. The evaluation recommends **proceeding carefully** with a phased approach.

## Recommendation

**Option C (Recommended): Implement SSE first as intermediate remote transport, then upgrade to Streamable HTTP.**

### Rationale

1. **SSE transport is available NOW** in the current SDK v0.5.0 → no upgrade needed.
2. **SSE is officially supported** by MCP protocol and many MCP clients.
3. **SSE and Streamable HTTP share the same tool handler code** — wire once, upgrade transport later.
4. **SDK v1.29.0 adds ~10+ heavy dependencies** (express, hono, jose, cors, express-rate-limit) that are unnecessary for the current scope.
5. **Zod version conflict** — current project uses `zod@^3.23.8`, SDK 1.29 requires `zod@^3.25`.
6. **`Server` class is deprecated** in v1.29 — migrating to `McpServer` would require refactoring all handler registrations.
7. **Lower risk** — SSE avoids SDK upgrade risk while still enabling remote MCP testing.

### Phase Plan

| Phase | Action | Risk |
|---|---|---|
| Phase 16 (this) | ✅ Evaluation complete | None |
| Phase 16a | Implement SSE transport (current SDK, no upgrade) | Low |
| Phase 16b | Upgrade to SDK v1.29.0 for Streamable HTTP | Medium |
| Phase 16c | Migrate from deprecated `Server` to `McpServer` API | Medium |

### Risk Level

- **SSE implementation (Phase 16a):** Low — existing SDK, existing patterns.
- **SDK upgrade (Phase 16b):** Medium — new dependencies, zod version change.
- **Full Streamable HTTP (Phase 16c):** Medium — requires SDK upgrade first.

### Cuan Insight Changes

## Security Notes

| Concern | Status |
|---|---|
| Auth token via `Authorization` header | ✅ Standard, sufficient |
| CORS needed | ✅ Yes, if browser-based clients access the HTTP endpoint |
| Public internet exposure | ⚠️ Only behind HTTPS + Bearer token + rate limiting |
| Cloudflare Tunnel / reverse proxy | ✅ Recommended for production remote MCP |
| Rate limiting | ✅ Recommended — use `express-rate-limit` or reverse proxy |
| Log redaction | ✅ `Authorization` and `providerToken` must never appear in logs |
| Request body logging | ⚠️ Must be redacted or disabled — could contain tokens |
| `providerToken` leak | ⚠️ Must not appear in error responses or MCP tool responses |
| Pre-existing security patterns in codebase | ✅ Redact functions exist: `redactErrorMessage`, `redactTokenLikeValues` |
